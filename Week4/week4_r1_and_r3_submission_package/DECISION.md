# DECISION.md — Week 4 Architecture Decision

## What did we choose to automate first?

For Sprint 4, we chose to automate the first repeatable market data-collection step instead of building a polished dashboard.

## Why this first?

The most painful manual step in the previous sprint was repeatedly collecting the same market evidence, saving screenshots, copying values, and converting them into a form that agents and LLMs could use. This task is repetitive, error-prone, and easy to inspect once automated. Automating the data fetch first gives the team a stable input layer for future Almanac, Macro, Technical, LLM, Human Score, and Delta Engine work.

## Why this matches the course roadmap

The Sprint 4 requirement is to ship something that runs, with at minimum one automated data-fetch script committed to GitHub. The roadmap expects teams to move from a manual pipeline toward structured and automated outputs. A small working script is better than a dashboard mockup because it creates real evidence and can be extended in later sprints.

## Definition of Done for this decision

The automation is done when:

1. The script runs from the repo.
2. It saves output as JSON or CSV.
3. The output file is committed or generated by the workflow.
4. README explains how to run it.
5. Every team member can explain what the script does.

## Product Owner note

This decision protects sprint value because it reduces manual data-entry friction and makes the prediction pipeline more inspectable. The system does not need to be impressive yet. It needs to be real, runnable, and useful for the next prediction.
